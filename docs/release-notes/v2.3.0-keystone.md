# Bazarr+ v2.3.0 (Keystone)

Bazarr+ v2.3.0 makes provider extensibility a first-class platform feature. Provider Hub V1 lets operators install provider plugins from catalog sources, stage updates safely, keep plugin secrets out of the UI, and activate changes on restart. Under that visible feature is a deeper foundation rebuild: Bazarr+ rebases its subtitle patch layer onto Subliminal 2.6, removes the old bundled dependency tree, and adds stronger dependency and provider compatibility checks.

53 commits since v2.2.1. The release touches the provider stack, dependency loading, Docker packaging, CI, and the Settings > Providers experience.

---

## Headline: Provider Hub

Provider Hub is the new plugin foundation for subtitle providers. It is designed for end users who want simple provider installation, while still giving Bazarr+ enough structure to treat third-party Python packages carefully.

What it does:

- **Catalog sources**: Bazarr+ ships with an official GitHub catalog source and supports additional GitHub catalog sources for community providers.
- **Marketplace UI**: Settings > Providers now includes a Provider Hub marketplace with search, source filters, trust badges, install actions, update actions, connection testing, and removal.
- **My Providers integration**: Installed plugins appear beside shipped providers, with source labels so users can see whether a provider is built in or installed through Provider Hub.
- **Staged lifecycle**: Installs, updates, and active removals are staged first. Bazarr+ keeps the current active provider running until restart, then promotes or removes the staged plugin.
- **Restart banner**: The UI shows a clear restart-required state when staged changes are waiting.
- **Activity log**: Catalog refreshes, installs, updates, tests, and removals are recorded so operators can see what happened and why.

Provider Hub does not hot-reload Python provider modules. That is intentional. Live unload and reload of provider code is risky, especially when third-party dependencies are involved. V1 favors restart activation because it gives Bazarr+ a stable boundary between the currently running provider set and the next one.

## Plugin Safety Model

Provider Hub's hard part is not the marketplace card. The hard part is the boundary around code Bazarr+ did not ship.

Every provider package must declare a manifest. Bazarr+ validates:

- Provider id, name, version, supported media, and languages.
- Entry module and entry class.
- Declared files and SHA-256 file hashes.
- Bundle SHA-256.
- GitHub source repository, commit, catalog URL, and optional source path.
- Exact, hash-checked Python dependencies.
- Secret fields that must be redacted before data reaches the UI.

Install-time behavior is guarded too:

- Provider files are fetched from a pinned GitHub commit.
- Bundles are verified before use.
- Dependencies install into a provider-specific virtual environment.
- Pip runs with hash checking and binary-only constraints.
- A worker process imports and smoke-validates the provider before the install is staged.
- Active plugins run through a worker protocol that serializes video, language, candidate, and download payloads across a clear boundary.

Trust is explicit. The official catalog source is trusted. Custom catalog sources are stored as untrusted, even if a caller tries to mark them trusted. That does not make community plugins unusable. It makes the trust signal honest.

## Subliminal 2.6 Foundation

v2.3.0 rebases the Bazarr+ subtitle patch layer onto Subliminal 2.6 and removes the legacy vendored dependency tree from the release path.

Why this matters:

- Bazarr+ can rely on current Python packages instead of carrying a large stale `libs/` snapshot.
- Dependency loading is now explicit and tested.
- Docker images install runtime dependencies through pip during the build.
- The patched provider layer stays in `custom_libs/`, where Bazarr+-specific behavior belongs.
- Provider compatibility guards cover the high-risk edges of the Subliminal rebase.

The test suite now includes dedicated checks for dependency loading, Subliminal version behavior, embedded subtitles, FileBot refinement, Subsunacs behavior, SignalR compatibility, and provider registration.

## Provider Fixes

Provider and search behavior also improved around the new foundation:

- Manual search now queries all enabled providers instead of stopping too early.
- Plugin providers are registered dynamically when active Provider Hub installations exist.
- Provider auth/config assembly includes active Provider Hub configuration and keeps secret values redacted from the UI.
- Shipped provider metadata now includes generated language coverage data for a richer picker experience.
- Prijevodi-online provider support is included from the hotfix line.

---

## Reliability

v2.3.0 includes several stability fixes that matter in real installations:

- Manual download cache misses caused by dual `sys.path` imports are fixed.
- Lingarr retries now wait long enough for cold-start deployments.
- Translate-only mass operations avoid an unnecessary full table join.
- SignalR dependency bumps are handled through a compatibility layer.
- Docker packaging includes the PostgreSQL requirements file in the image build.
- The CI backend job installs ffmpeg so embedded-subtitle tests exercise real media fixtures.

---

## UI / UX

Settings > Providers received the largest UI change in this release.

- Provider Hub marketplace with catalog cards, update cards, source drawers, empty states, and status badges.
- Trust badges for official versus community catalog entries.
- A staged-change banner that explains when a restart is required.
- A compact `Pending restart` state so narrow provider cards do not clip actions.
- Plugin/shipped labels in the provider picker and provider list.
- Generated provider language metadata for clearer provider selection.

The goal is practical: installing a provider should feel like installing a plugin, not editing source code.

## CI / Docker

- CI now tests Python 3.12, 3.13, and 3.14 backend paths with the dependency and Subliminal rebase guards.
- Frontend CI continues to typecheck, lint, format-check, test, and build the React UI.
- Docker release builds publish versioned, `latest`, and SHA tags only from release-tagged flows.
- GitHub Pages deployment for the static site is separated from Docker release publishing.

## Migration from v2.2.1

Docker users can upgrade in place.

```bash
docker pull ghcr.io/lavx/bazarr:2.3.0
# or :latest after the release workflow publishes
docker compose up -d
```

Provider Hub is additive. Existing shipped providers and provider settings continue to work. Installed Provider Hub plugins only search after they are added to the enabled provider list and settings are saved.

For source installs, refresh Python dependencies after pulling this release. v2.3.0 no longer relies on the old checked-in `libs/` dependency tree.
